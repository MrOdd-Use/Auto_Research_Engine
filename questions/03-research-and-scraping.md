### Q08（基础）：Web Research / RAG / Tool-use 的关系是什么？你们的 grounding 约束是什么？

**标准答案（可直接说）：**
一句话：Tool-use 是调用外部能力（搜索/抓取）；RAG 是把外部证据注入上下文；Web Research 是面向互联网证据的 RAG/Tool-use 组合。本项目通过“证据采集 + 校验 gate + 引用输出约束 + 评测”来做 grounding。

展开：研究阶段会通过检索器拿到候选 URL，再抓取正文形成证据语料，经过去重与重排后进入写作/总结。除传统 web 搜索外，本项目也支持通过 MCP（Model Context Protocol）把“外部工具/数据源”以统一协议接入检索链路（例如内部知识系统/数据库/本地工具），并将工具返回标准化成与 web 搜索一致的结果结构供后续复用（见 `questions/07-mcp-integration.md`）。多智能体链路里，Check Data 作为 gate 用硬约束（主体/时间/指标等）评估证据是否匹配，决定 ACCEPT/RETRY/BLOCKED，避免“证据不足仍强行生成”。离线 hallucination eval 则把输出与 source text 做对齐式评估，作为质量信号。

常见坑/反杀点：
- grounding 不是“加链接就行”；真正难点是断言是否被证据支持。
- Web 场景有 prompt injection/SEO spam 风险，生产级需要更多防护（优化里讲）。

**相关模块（对应实现）：**
- gpt_researcher/actions/query_processing.py
- gpt_researcher/actions/web_scraping.py
- gpt_researcher/retrievers/mcp/
- gpt_researcher/mcp/
- evals/hallucination_eval/

**技术细节（实现 / 为什么 / 利弊）：**
- 实现：Web 证据路径是“检索→抓取→片段选择→写作”，ASA 流在 scrap 模块内完成多引擎检索、URL 去重、抓取物化与 MMR 选片（见 `multi_agents/agents/scrap.py`）；写作阶段要求输出带链接与 sources 列表（见 `multi_agents/agents/writer.py`）。
- 为什么：RAG/Tool-use 的核心不是“有链接”，而是让关键断言尽量落到可追溯证据片段，并在证据不足时触发回流或降级。
- 利弊：利是显著降低“想当然”；弊是依赖抓取质量与网页可访问性，且当前实现仍是段落级/启发式门禁，非严格断言级对齐。

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
> **举例提示（面试官听不懂就用）：** 把本题映射到“AI 对就业市场的影响”示例卡片来讲——先追问口径 → 产出结构化大纲（`section_details`）→ 按章生成 `research_queries` → 证据采集与门禁 → 写作/审阅/发布。这样抽象概念会变成“我在这个例子里具体做了什么”。
> 本题映射：示例里所有“影响/趋势/建议”都要能对应到证据片段；证据不足就降级措辞并写清时间窗与不确定性，而不是强行下结论。

**定义：** grounding 的目标是什么？  
答题要点：关键断言可被外部证据验证，降低幻觉与“想当然”。

**实现：** 你们怎么把 web 证据变成上下文？  
答题要点：检索→抓取→去重/选择→写作；多轮采集中用重排提高证据密度。

**边界：** 哪些信息无法被 grounding？  
答题要点：无公开来源、强实时数据、不可访问来源；需要输出不确定或阻断。

**取舍：** 为什么先 gate 再写作？  
答题要点：尽早暴露证据不足，避免在错误证据上投入写作与审阅成本。

**优化：** 如何升级到断言级对齐？  
答题要点：抽取 claims、做 claim-evidence alignment、把未支撑断言回流补检索（可选升级）。

### Q09（基础）：URL 归一化去重为什么关键？不做会造成什么系统性问题？

**标准答案（可直接说）：**
一句话：URL 归一化去重能避免同一页面以不同参数/锚点重复出现，减少重复抓取、重复证据与上下文污染。

展开：Web 检索结果常见“同内容多 URL”（不同 query 参数、尾部斜杠、fragment 等）。不去重会导致抓取资源浪费、证据片段重复、重排与写作偏向单一来源，从而降低覆盖度并增加幻觉风险。本项目在证据采集阶段对 URL 做归一化，并以归一化 key 去重后再抓取物化。

常见坑/反杀点：
- 只说“省资源”；更要强调“避免证据同质化导致结论偏差”。
- 去重过强可能误杀“同域不同页面”；需要粒度控制（优化）。

**相关模块（对应实现）：**
- multi_agents/agents/scrap.py

**技术细节（实现 / 为什么 / 利弊）：**
- 实现：scrap 模块对 URL 做归一化（去 fragment、排序 query 参数、去尾部斜杠）再去重，并用归一化 key 把抓取结果对齐回搜索结果（见 `multi_agents/agents/scrap.py` 的 URL 归一化与去重流程）。
- 为什么：避免重复抓取/重复证据造成上下文同质化，进而让写作偏向单一来源。
- 利弊：利是节省抓取预算并提升多样性；弊是归一化过强可能把“语义不同但 URL 相近”的页面合并，需要按站点特征做例外规则（可选升级）。

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
> **举例提示（面试官听不懂就用）：** 把本题映射到“AI 对就业市场的影响”示例卡片来讲——先追问口径 → 产出结构化大纲（`section_details`）→ 按章生成 `research_queries` → 证据采集与门禁 → 写作/审阅/发布。这样抽象概念会变成“我在这个例子里具体做了什么”。

**定义：** 什么叫 URL 归一化？  
答题要点：去掉 fragment、规范 path、对 query 参数排序与规范化，得到稳定 key。

**实现：** 归一化后怎么去重？  
答题要点：以归一化 key 存入映射结构，只保留一条代表结果，再进入抓取。

**边界：** 哪些站点归一化容易出错？  
答题要点：query 参数承载关键内容（分页/地区/语言）；需要更谨慎策略（可选）。

**取舍：** 为什么不只做“正文哈希去重”？  
答题要点：正文哈希需要先抓取；URL 去重能在抓取前就降成本，两者可组合（优化）。

**优化：** 生产级怎么做更稳？  
答题要点：URL 去重 + 正文近似去重（simhash/embedding）+ 来源多样性约束（可选升级）。

### Q10（基础）：静态抓取 vs 动态渲染抓取（JS 页面），各自优缺点与触发条件是什么？

**标准答案（可直接说）：**
一句话：静态抓取（HTML 解析）成本低、速度快；动态渲染抓取（浏览器/JS 执行）覆盖更广但更慢更重，适合作为兜底策略。

展开：静态抓取适合内容在初始 HTML 就完整可见的站点；动态渲染适合 SPA/强依赖 JS 的站点或静态拿不到正文的场景。仓库具备静态抓取链路，并有可选的浏览器抓取依赖（selenium/playwright）。工程实践通常采用“静态优先、动态兜底”，并配合限流、缓存与失败重试（部分属于可选升级）。

常见坑/反杀点：
- 动态抓取不是越多越好：成本、稳定性与封禁风险更高。
- 面试官常追问验证码/登录墙；当前仓库未完整覆盖，需按优化口径回答。

**相关模块（对应实现）：**
- gpt_researcher/actions/web_scraping.py
- gpt_researcher/scraper/
- pyproject.toml

**技术细节（实现 / 为什么 / 利弊）：**
- 实现：抓取组件提供多种 scraper：静态解析（如 BeautifulSoup）与浏览器渲染抓取（Selenium/NoDriver 等）（见 `gpt_researcher/scraper/*`），由配置 `cfg.scraper` 选择。
- 为什么：静态抓取便宜且稳定；动态渲染用于 JS-heavy 页面（内容需执行脚本/滚动加载）。
- 利弊：静态优点是快/资源少但易抓不到；动态优点是覆盖更多页面但慢、依赖浏览器环境且更容易受反爬/登录墙影响。

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
> **举例提示（面试官听不懂就用）：** 把本题映射到“AI 对就业市场的影响”示例卡片来讲——先追问口径 → 产出结构化大纲（`section_details`）→ 按章生成 `research_queries` → 证据采集与门禁 → 写作/审阅/发布。这样抽象概念会变成“我在这个例子里具体做了什么”。

**定义：** 动态渲染抓取解决了什么？  
答题要点：拿到 JS 渲染后的 DOM 与正文，覆盖 SPA/懒加载内容。

**实现：** 抓取产物在系统里怎么用？  
答题要点：形成 `raw_content` 与元数据，供后续证据选择、校验与写作消费。

**边界：** 哪些页面即使动态渲染也难抓？  
答题要点：强反爬、验证码、登录墙、付费墙；需要代理池/账号体系（可选升级）。

**取舍：** 为什么不全部用浏览器抓取？  
答题要点：吞吐与成本不可控；静态优先能覆盖大多数页面。

**优化：** 如何提升抓取成功率与质量？  
答题要点：站点分层策略、浏览器池、内容抽取质量评估、缓存与重试（可选升级）。

### Q11（基础）：什么是 MMR？它在证据选择里解决了什么矛盾？

**标准答案（可直接说）：**
一句话：MMR（Maximal Marginal Relevance）是在“相关性”和“多样性”之间折中，避免 top-k 全是同质信息。

展开：只按相关性排序会选出大量相似片段，导致覆盖度差、结论偏单一来源。MMR 用“对 query 的相关性”减去“与已选内容的相似度惩罚”，在有限上下文预算下最大化信息增益。本项目在 ASA 证据采集阶段对候选 passages 做 MMR 选择，从而提升覆盖与多样性。

常见坑/反杀点：
- MMR 不是简单去重，而是选择策略（re-ranking / selection）。
- 面试官会追问相似度计算与参数选择；要讲清当前实现与升级路径。

**相关模块（对应实现）：**
- multi_agents/agents/scrap.py

**技术细节（实现 / 为什么 / 利弊）：**
- 实现：MMR 在 scrap 模块内优先用 embedding 向量做余弦相似度做“相关性 - 多样性惩罚”的迭代选 top-k；若 embedding 不可用/超时则回退到轻量词袋计数向量（见 `multi_agents/agents/scrap.py`）。
- 为什么：在有限上下文预算里，既要相关又要避免同质片段堆叠，提高信息增益。
- 利弊：embedding 语义更强但有成本/依赖/失败风险；词袋回退便宜快且更稳，但会忽略同义改写与跨语表达。

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
> **举例提示（面试官听不懂就用）：** 把本题映射到“AI 对就业市场的影响”示例卡片来讲——先追问口径 → 产出结构化大纲（`section_details`）→ 按章生成 `research_queries` → 证据采集与门禁 → 写作/审阅/发布。这样抽象概念会变成“我在这个例子里具体做了什么”。

**定义：** MMR 的核心目标是什么？  
答题要点：top-k 同时高相关、低冗余，提升信息增益。

**实现：** 你们怎么表示文本向量与相似度？  
答题要点：优先用 embedding 向量 + 余弦相似度；失败/超时回退到轻量词袋/计数向量。

**边界：** MMR 在什么情况下效果一般？  
答题要点：候选内容整体质量差或高度同质；query 太宽泛导致相关性不稳定。

**取舍：** 为什么还要保留词袋回退？  
答题要点：embedding 调用可能超时/限流/缺配置；回退保证在线链路不被卡死。

**优化：** 如何更“证据友好”？  
答题要点：加入来源多样性硬约束、发布时间/权威性权重、近似去重与聚类（可选升级）。

### Q12（基础）：为什么要做 ASA 多轮证据采集？单轮检索常见失败在哪里？

**标准答案（可直接说）：**
一句话：多轮证据采集通过“目标分解 + 引擎扩展 + 去重重排 + 校验回流”解决首轮检索不全、口径不对、证据同质化的问题。

展开：单轮检索容易被 SEO 垃圾、单一来源或错误口径带偏；对复杂主题往往覆盖不足。本项目的 ASA loop 会把章节主题分解为多个独立搜索目标；不同轮次选择不同检索引擎组合，并对纵向引擎失败做回退；对结果去重、抓取物化与 MMR 重排；并根据校验反馈决定是否进入下一轮，从而提高覆盖与证据质量。

常见坑/反杀点：
- 多轮不是无限轮：要有上限与停止条件。
- 迭代依据要能落到“校验反馈/置信度/失败类型”等信号。

**相关模块（对应实现）：**
- multi_agents/agents/scrap.py
- multi_agents/agents/check_data.py

**技术细节（实现 / 为什么 / 利弊）：**
- 实现：ASA 通过“目标分解（每轮 2-4 个 search targets）→多引擎检索→垂直引擎失败回退→URL 去重→抓取物化→MMR 选片”多轮执行，轮次数由 `audit_feedback` 置信度驱动并受 `scrap_max_iterations` 上限控制（见 `multi_agents/agents/scrap.py`）。
- 为什么：单轮检索常被 SEO 噪声/单一来源/口径不对带偏；多轮允许根据缺口调整约束与引擎组合。
- 利弊：利是覆盖与证据质量提升；弊是成本/时延增加且有收益递减，需要停止条件与缓存（可选升级）。

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
> **举例提示（面试官听不懂就用）：** 把本题映射到“AI 对就业市场的影响”示例卡片来讲——先追问口径 → 产出结构化大纲（`section_details`）→ 按章生成 `research_queries` → 证据采集与门禁 → 写作/审阅/发布。这样抽象概念会变成“我在这个例子里具体做了什么”。

**定义：** “Adaptive”体现在哪里？  
答题要点：轮次策略变化（引擎组合、模型档位、目标分解），并受校验反馈影响。

**实现：** 每轮核心步骤？  
答题要点：目标分解→多引擎检索→URL 去重→抓取物化→passage 构建→MMR 选择→形成证据包。

**边界：** 哪些主题即使多轮也难收敛？  
答题要点：缺少公开资料、信息高度冲突且无权威来源、强时效实时数据。

**取舍：** 为什么不直接扩大单轮 top-k？  
答题要点：引入更多噪声与同质内容；多轮允许定向扩展与质量控制。

**优化：** 如何更快收敛？  
答题要点：缓存与增量检索、结构化缺口清单驱动查询、失败类型分流策略（可选升级）。

### Q15（基础）：什么是上下文工程（context engineering）？你们主要解决哪几类问题？

**标准答案（可直接说）：**
一句话：上下文工程是在有限上下文窗口下做信息选择、压缩、去噪与结构化输入；本项目重点解决证据噪声、重复、覆盖不足与 token 预算。

展开：原始网页内容长且噪声大，直接塞入上下文会导致注意力分散与幻觉风险上升。本项目在证据采集侧通过去重与 MMR 做“高信息密度选择”，在研究系统侧也提供基于 embeddings 的相似内容检索与压缩能力，把上下文从“堆网页”变成“精选证据片段 + 结构化提示”。

常见坑/反杀点：
- 上下文工程不是简单截断；关键是选择策略与结构化组织。
- 面试官会追问“如何避免丢关键证据”；要强调多样性与覆盖策略。

**相关模块（对应实现）：**
- multi_agents/agents/scrap.py
- gpt_researcher/skills/context_manager.py
- gpt_researcher/memory/embeddings.py

**技术细节（实现 / 为什么 / 利弊）：**
- 实现：上下文压缩使用“切分→embedding 相似度过滤→返回 top chunks”的压缩检索链，阈值由 `SIMILARITY_THRESHOLD` 控制（见 `gpt_researcher/context/compression.py`）；ASA 侧用 MMR 选片提升证据密度（见 `multi_agents/agents/scrap.py`）。
- 为什么：Web 抓取内容噪声大且 token 预算有限，需要把上下文变成“高密度证据片段”。
- 利弊：利是更省 token、降低噪声；弊是压缩可能误删关键反证/限定条件，需要在评测与回归集中覆盖失败样本。

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
> **举例提示（面试官听不懂就用）：** 把本题映射到“AI 对就业市场的影响”示例卡片来讲——先追问口径 → 产出结构化大纲（`section_details`）→ 按章生成 `research_queries` → 证据采集与门禁 → 写作/审阅/发布。这样抽象概念会变成“我在这个例子里具体做了什么”。

**定义：** 上下文工程的目标？  
答题要点：在固定预算下最大化可用证据与可控性，降低噪声与重复。

**实现：** 你们用了哪些手段？  
答题要点：证据侧去重+MMR；研究侧相似内容检索与压缩；规划信息结构化注入提示。

**边界：** 仍可能失败的原因？  
答题要点：证据本身质量差、主题过宽、权威来源缺失、压缩丢限定词。

**取舍：** 为什么不完全依赖向量库？  
答题要点：Web 场景仍需要抓取与清洗；向量库更适合内网/长期记忆，属于可扩展方向。

答题要点：断言级引用跨度对齐、来源可信度模型、反证检索、证据图组织、缓存与增量更新（可选升级）。

### Q25（复杂）：ASA 多轮采集的整体策略是什么？（目标分解、引擎选择、回退、去重、物化抓取、证据片段构建）

**标准答案（可直接说）：**
一句话：ASA 把章节主题拆成多个独立搜索目标，按轮次与领域选择检索引擎并回退，先 URL 去重再抓取物化，最后用 MMR 选出多样且高相关的 top 证据片段形成证据包。

展开：首轮通常追求快速覆盖；后续轮次按主题领域扩展到更专业的引擎组合，并对纵向引擎失败做通用引擎回退。检索结果先 URL 归一化去重，减少抓取成本并避免证据同质；随后抓取正文并提取元数据，构造成 passages（content + source_url + metadata）。对 passages 做重排选择得到 top 证据，并将轮次、引擎、目标与 top 证据片段记录为证据包，供校验与写作使用。

常见坑/反杀点：
- 只讲多轮不讲“轮次差异”；要说明领域与回退策略。
- 参数（目标数、top-k）是工程折中，面试要能解释为可调。

**相关模块（对应实现）：**
- multi_agents/agents/scrap.py
- gpt_researcher/actions/web_scraping.py

**技术细节（实现 / 为什么 / 利弊）：**
- 实现：ASA 的关键策略在 scrap 模块：每轮把章节主题分解成 2-4 个独立 targets（JSON list），按轮次选择引擎（首轮 tavily，后续按域选引擎），垂直引擎失败回退到 google/bing，随后去重→抓取→构建 passages→MMR 选 top 片段，形成 `scrap_packet.search_log`（见 `multi_agents/agents/scrap.py`）。
- 补充：什么是「垂直引擎失败回退」？指 arxiv / semantic_scholar / pubmed_central 这类领域引擎在“调用失败/不可用”时，自动用通用引擎（google/bing）对同一 target 再检索以兜底覆盖；注意当前实现按“failed=True”触发（并非仅因结果为空就回退）。
- 为什么：把“搜什么”从单 query 扩展成可控的目标集合，并用回退与重排提升覆盖/多样性。
- 利弊：利是鲁棒性更好；弊是对 prompt/引擎配置更敏感，需要限流、缓存与失败类型分流（可选升级）。

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
> **举例提示（面试官听不懂就用）：** 把本题映射到“AI 对就业市场的影响”示例卡片来讲——先追问口径 → 产出结构化大纲（`section_details`）→ 按章生成 `research_queries` → 证据采集与门禁 → 写作/审阅/发布。这样抽象概念会变成“我在这个例子里具体做了什么”。

### Q26（复杂）：为什么 MMR 默认用 embedding，但仍保留词袋/时间预算回退？

### Q26a（复杂）：从“搜索 → 抓取 → 重排 → 上下文压缩”的全链路怎么跑？为什么这样设计？

**标准答案（可直接说）：**
一句话：先用搜索找到候选 URL，再对网页做去重与正文抓取，把正文切成片段后用“相关性 + 多样性”重排选出证据包，最后用 embedding 相似度过滤把上下文压缩到可喂给模型的长度。

展开（按顺序讲清楚每一步在解决什么）：
1) **搜索阶段（找“哪里有信息”）**：先拿到链接、标题和摘要（snippet）。这一步快，适合做覆盖。
2) **去重阶段（省预算、控偏差）**：同一页面常以不同 URL 形态出现（参数/锚点/尾斜杠等），先归一化去重，避免重复抓取与证据同质化。
3) **抓取阶段（把 URL 变成“可用正文”）**：对去重后的 URL 批量抓取正文文本（raw content）。抓取有并发与限流；按页面类型选择抓取方式（例如 PDF 提取、静态 HTML 解析、必要时浏览器渲染作为兜底）。抓取失败或正文过短时，工程上通常要么跳过该源，要么降级使用摘要作为临时线索（但摘要不如正文可引用）。
4) **片段构建（把长文切成可选单元）**：把正文按句子/长度切成多个 passages，保证每个片段既足够信息密度又不会太长。
5) **重排选片（解决“相关 vs 同质”的矛盾）**：对候选 passages 做重排，优先保留与查询最相关、且彼此不重复的 top-k 片段；设置时间预算，超时宁可降级到更简单排序，也不阻断在线链路。
6) **上下文压缩（把证据缩到 prompt 能承载）**：把抓取到的文档再切成更小的 chunks，用 embedding 相似度过滤掉与问题无关的部分，只保留最相关的若干块作为最终上下文（可附带来源信息）。

常见坑/反杀点：
- 把“搜索摘要”当“网页正文”；要强调摘要只是线索，正文才是可引用证据。
- 只讲抓取不讲去重与重排；面试官会追问“怎么避免同质证据堆叠”。
- 只讲 embedding 不讲成本/超时/限流；要说清楚降级策略与延迟上界。

**相关模块（对应实现）：**
- multi_agents/agents/scrap.py（搜索→去重→抓取物化→passage 构建→MMR 重排）
- gpt_researcher/scraper/（多种抓取后端：静态/浏览器/PDF 等）
- gpt_researcher/context/compression.py（chunk 切分 + embedding 相似度过滤的上下文压缩）

**追问链（定义 → 实现 → 边界 → 取舍 → 优化）：**
**定义：** “物化抓取 / 重排 / 压缩”分别在做什么？  
答题要点：物化=URL→正文；重排=相关+多样性选 top-k；压缩=只保留与问题强相关的 chunks。  

**边界：** 哪些情况抓取与压缩仍会失败？  
答题要点：强反爬/验证码/登录墙/JS 复杂渲染/内容极短；embedding 配额/超时；需要兜底与人工复核。  

**取舍：** 为什么不“抓完全文再让模型自己挑”？  
答题要点：上下文与成本不可控；先选片/压缩能把预算花在最相关信息上，并减少噪声导致的幻觉。  

**优化：** 生产级如何更稳更省？  
答题要点：抓取缓存、近似去重（内容级）、两阶段重排（轻量粗排+语义精排）、来源多样性约束与权威性权重（可选升级）。

### 7) URL 归一化去重怎么做？为什么能减少系统性偏差？
**一句话结论：** 先把“同页多 URL”归一化成稳定 key，再去重与对齐抓取结果，避免重复证据污染上下文。

**实现落点：**
- `multi_agents/agents/scrap.py`：URL 归一化规则：去 fragment、query 参数排序、去尾部斜杠、空 path 归 `/`；以归一化后的 URL 作为 dedupe key。
- 去重发生在“抓取物化之前”，减少重复抓取与重复片段进入 MMR。
- `tests/test_scrap_agent.py`：`test_url_set_union_dedup` 覆盖不同参数顺序/尾斜杠的去重结果。

**为什么这样设计：**
- 重复抓取会导致证据片段高度同质，MMR 与写作会被单一来源“绑架”，形成系统性偏差。

**利弊与边界：**
- 利：省抓取预算、提高证据多样性。
- 弊：归一化过强可能误合并“看似同页但内容不同”的 URL（例如分页/地域参数），需要站点级例外（可选升级）。

**追问补充：URL 归一化/去重/物化抓取的典型坑与 fallback（贴合当前实现）**
- redirect：当前去重 key 以“归一化后的 URL”作为 map key；若抓取返回的最终 URL 与原 URL 不一致，可能出现“抓取结果对不上去重 key/命中失败”的情况，导致漏物化或重复物化。
- tracking params：目前只对 query 参数做排序，不做参数白名单/黑名单剔除；同内容多 URL（带各种 tracking）仍可能绕过去重。
- 同源不同路径：只做 URL 层 dedupe，不做 content-level canonical（可选升级：内容哈希/主句去重/正文相似度聚类，未实现）。
- 抓取失败 fallback：`raw_content` 为空会被丢弃（即使搜索摘要 `body` 有内容），可能降低 recall；可选升级：保留 snippet 兜底并标注可信度（未实现）。

**如何验证：**
- `tests/test_scrap_agent.py` 的 URL 去重用例可直接回归。

### 8) MMR 的实现细节（向量/相似度/lambda/top_k/time budget）与取舍？
**一句话结论：** 证据片段重排采用“双路径 MMR”：优先走 embedding-MMR（更语义，受 time budget 与候选裁剪控制），失败/超时自动降级到词袋（BOW）MMR，确保在线链路稳定输出 top-10 多样且相关的片段。

**实现落点：**
- `multi_agents/agents/scrap.py`：MMR 重排入口默认 `top_k=10`、`lambda_param=0.7`、`time_budget_s=5.0`；当 `SCRAP_MMR_USE_EMBEDDINGS` 启用时，优先尝试 embedding-MMR，成功就直接返回。
- embedding-MMR 路径（可选）：会把总时间预算 `time_budget_s` 换算成“剩余时间”并用 `asyncio.wait_for(..., timeout=remaining)` 包住 embedding 调用；当 passages 过多时，用 `SCRAP_MMR_MAX_EMBED_PASSAGES`（默认 80）先基于轻量相关性粗排截断候选，再对候选做 embedding-MMR 精排。
- 自动降级策略：embedding 初始化缺配置（embedding provider/model）或 embedding 调用异常/超时，会在本进程内关闭 embedding-MMR 路径并记录错误，后续直接走 BOW-MMR，避免反复失败拖慢热路径。
- BOW-MMR 路径（兜底）：向量为词袋计数（英文 token + 中文单字），相似度为余弦；按 MMR 贪心策略在“相关性 vs 多样性”间折中选 top-k。若循环超过 `time_budget_s`，回退到“按相关性补齐”的兜底排序，防止在线链路卡死。
- `tests/test_scrap_agent.py`：`test_passage_mmr_top10_diversity` 覆盖 top-10 多样性结果（不重复 content）。

**为什么这样设计：**
- 证据选择是热路径：embedding-MMR 在同义改写/跨语言场景更鲁棒，但成本更高且依赖 embedding 配置/调用稳定性；因此采用“能用则用 embedding，不能用就退回词袋”的分层策略，把语义收益与工程稳定性同时兼顾。
- 候选裁剪 + time budget 是关键工程约束：保证在 evidence 数量暴涨时也能把最慢部分（embedding）限制在可控区间内。

**利弊与边界：**
- 利：在 embedding 可用时提升语义召回与去同质化效果；在 embedding 不可用/不稳定时依然能用纯本地计算（BOW）维持稳定时延与输出契约。
- 弊：embedding 路径依赖配置与外部调用（若用远程 embedding）；BOW 路径对同义改写/跨语言语义相近更容易退化。可选升级方向包括：embedding 结果缓存、跨轮证据缓存、以及更强的近重复检测（内容级 canonical）。

**追问补充：lambda 怎么调？效果如何评估？**
- 现状：`lambda_param` 目前是固定默认值（0.7），仓库内没有在线 A/B。
- 评估口径（可做但不宣称已做）：用 `evals/` 做离线对照实验（固定样本与配置，只改 lambda/是否启用 MMR），关注指标如：
  - 多样性/去重率：top-k 片段的近重复比例、来源域名数、URL 覆盖数。
  - 覆盖关键点比例：对 planner `key_points` 的覆盖度（规则或 judge 打分）。
  - hallucination 对齐分：长文输出与 source text 的对齐式评估（`evals/hallucination_eval`）。
