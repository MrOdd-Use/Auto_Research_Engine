## 📄 OpenCompass 动态模型梯队系统 (Multi-Agent Tier System)

### 1. 架构总览：分布式“军火库”

系统核心不再是一个中央决策者，而是由 **DataHub (数据枢纽)** 为每个 Agent 初始化的专属模型阶梯。

- **独立性：** Planner 的 Tier-0 不一定等于 Writer 的 Tier-0，取决于各自 Rerank 的维度偏好。
    
- **状态隔离：** `StateController` 为每个 `agent_id` 维护独立的 `current_idx`。
    

---

### 2. 初始化与路由逻辑

#### 2.1 启动序列 (Startup Sequence)

1. **缓存校验：** `DataHub` 检查本地 OpenCompass 数据。若过期（>30天），强制拉取。
    
2. **供应商过滤：** 提取环境变量中每个供应商最新的 3 个 LLM。
    
3. **专属 Rerank：** 针对每一个 Agent，调用 Gemini 推理模型，根据其描述（如推理、编程、事实性）对模型池重排，生成前 4 名（Top 4 Tier List）。
    
4. **持久化：** 结果存入 `agent_state.json`。
    

#### 2.2 静态权重映射

无论哪个 Agent，计算分数时均采用固定的权重阶梯：

$$FinalScore = 0.4 \times S_1 + 0.3 \times S_2 + 0.2 \times S_3 + 0.1 \times S_4$$

_注：$S_{1...4}$ 的维度顺序由 Gemini 针对不同 Agent 的描述动态决定。_

---

### 3. Agent 进化逻辑全景图

|**Agent 角色**|**核心关键词 (Rerank 依据)**|**初始档位**|**升档 (Promote) 触发条件**|**降档 (Demote) 触发条件**|
|---|---|---|---|---|
|**Planner**|推理, 规划, 对话, 记忆|**Tier-3**|1. 交互次数 $\ge 3$ 或 $6$；<br><br>  <br><br>2. 用户表达负面反馈（如“不对”）|任务确认并进入执行阶段|
|**Scrap**|摘要, 指令遵循, 长上下文|**Tier-3**|**基于轮次**：Round 1 (T3) $\rightarrow$ Round 2 (T2) $\rightarrow$ Round 3 (T1)|任务完成/采集成功|
|**Check Data**|逻辑, 事实性, NLI|**Tier-2**|1. 深度评估 (DeepEval)；<br><br>  <br><br>2. 识别到“模糊性 (Ambiguous)”关键词|执行简单原子化拆解 (SLM 模式)|
|**Writer**|写作, 语言, 风格化|**Tier-1**|1. 被 Review 驳回次数 $>1$；<br><br>  <br><br>2. 输入语料置信度 $<0.4$|仅处理简单数据罗列任务|
|**Review**|批判思维, 细节, 合规|**Tier-0**|**固定高档位**：通常锁定在 Tier-0/1|仅检查 Markdown 格式或错别字|
|**Revise**|格式保持, 指令遵循, 纠错|**Tier-2**|1. 循环报错 (修改无效)；<br><br>  <br><br>2. 收到“整体重构”类宏观指令|处理简单的局部数据替换|

### 4. 状态控制器 (StateController) 逻辑

系统通过 `StateController` 实现“按需换档”。

- **读取：** `get_current_model(agent_id)` 确保业务逻辑与具体的 Model ID 解耦。
    
- **原子更新：** 升/降档操作必须同步写入持久化存储，防止进程重启后丢失当前的进化等级。
    

---

### 5. 异常处理：维度不匹配保护

> **场景：** 当 OpenCompass 接口更新，返回了 5 个评测维度，而我们的静态权重数组只有 4 位。

1. **熔断：** 系统停止该 Agent 的 Rerank 过程。
    
2. **存证：** 自动创建 `debug/mismatch_[agent_id].json`。
    
3. **内容：** 记录 `agent_desc`、`expected_weights: 4`、`received_dims: [A, B, C, D, E]`。
    
4. **降级：** 恢复到系统默认的通用模型（如 `gpt-4o-mini`），确保全链路不中断。

### 6. 核心 Python 实现框架 (补充)

Python

```
class StateController:
    """管理每个 Agent 的档位变速箱"""
    def __init__(self, state_file="agent_state.json"):
        self.state_file = state_file
        self.states = self._load_states()

    def promote(self, agent_id):
        """向上升档：追求更强的逻辑能力"""
        if self.states[agent_id]['current_idx'] > 0:
            self.states[agent_id]['current_idx'] -= 1
            self._save_state()
            # 记录日志供后续分析
            logging.info(f"Agent {agent_id} promoted to better model.")

    def demote(self, agent_id):
        """向下降档：追求性价比与响应速度"""
        if self.states[agent_id]['current_idx'] < 3:
            self.states[agent_id]['current_idx'] += 1
            self._save_state()

    def set_tier(self, agent_id, tier_idx):
        """强制指定档位：应对特定复杂任务"""
        if 0 <= tier_idx <= 3:
            self.states[agent_id]['current_idx'] = tier_idx
            self._save_state()
```

# 产品需求文档：语音驱动型 Planner Agent 

## 1. 产品愿景

用户只需通过口头交流或简短文字，即可将脑中的模糊意图转化为多维度的专业计划。系统通过跟踪交互深度，自动升级底层的逻辑推理能力。

---

## 2. 核心交互流程 (The Audio-First Flow)

1. **需求投喂 (Input)：** 用户通过自然语言描述目标。
    
2. **多维解构 (Decomposition)：** Agent 语音播报（或简要文本）核心框架，同步在看板生成大纲。
    
3. **对话微调 (Dialogue Refinement)：** 用户通过对话修正（如：“把第二阶段拉长两周”、“考虑到预算减半”）。
    
4. **无感升阶 (Silent Escalation)：** 交互达到 3 次，Agent 的语气和分析深度会产生质变，从“执行者”变为“策略咨询师”。
    

---

## 3. 功能模块需求

### 3.1 语义拆解引擎 (Language Parser)

- **多视角识别：** 听到 Query 后，自动提取任务的**时间（Time）**、**视角（Role）**、**依赖关系（Dependency）**。
    
- **默认输出：** 每次调整后，Agent 会先口头确认：“已为您调整了 [X]，现在的逻辑是 [Y]，这样可以吗？”
    

### 3.2 动态升阶算法

- **L1 (0-2次交互)：** 响应极快，重在记录和生成基础列表。
    
- **L2 (3-5次交互)：** 触发模型升阶。Agent 会开始说：“我发现之前的调整让时间线变得很紧，建议我们在 [A] 环节增加缓冲。”（体现推理能力）。
    
- **L3 (6次以上)：** 顶级模型介入。Agent 能够处理全局冲突，并能反向质疑用户逻辑中不合理的地方。
    

### 3.3 极简 UI 看板 (配套视觉)

- **极简列表：** 仅显示当前任务的树状结构。
    
- **状态波形：** 语音交互时显示的波动条，颜色随模型等级变化（Lv.1 蓝色 -> Lv.2 紫色 -> Lv.3 金色）。
    
- **修改计数器：** 屏幕角落显示 `Refinement: 3/6`。
    

---

## 4. 核心逻辑规则

|**维度**|**处理方式**|
|---|---|
|**修改判定**|只有当用户提出具体的“增加/删除/修改/细化”指令时，计数器 +1。简单的“好的”、“谢谢”不计数。|
|**冲突管理**|如果用户指令与前文矛盾，Agent 需在语音中明确提示：“这与您之前确定的 [节点A] 有冲突，需要我帮您重新平衡吗？”|
|**上下文继承**|升阶时，系统自动汇总前 3 次修改的内核，作为新模型的“元指令（Meta-Prompt）”。|

---

## 5. 交互示例 (User Experience Case)

- **User:** “我想做一个出海东南亚的市场计划。”
    
- **Agent (Lv.1):** “没问题。我从时间、渠道、合规三个维度为您拆解了大纲，请看屏幕。”
    
- **User:** “把新加坡作为第一站，去掉越南部分。” (Count 1)

- **Agent (Lv.2 - 升阶):** “**由于预算和周期大幅缩减，增加 KOL 环节可能会导致执行过载。** 我建议将 KOL 合作集中在预热期，并已为您优化了成本分配，您看这样调整合理吗？”

## 6. 技术约束

- **Latency (延迟)：** 语音响应延迟控制在 **1.2s** 以内。
    
- **Consistency (一致性)：** 升级模型后，严禁丢失用户在 Lv.1 阶段提出的具体细节。

# Scrap Agent 详细需求文档 (PRD)

## 1. 产品定义

**Scrap Agent** 是多智能体系统中的执行单元。其核心任务是根据初始查询或审计反馈，动态调度搜索引擎，提取、压缩并精炼出 **Top 10 不重复且具有多样性** 的客观文段。它不负责判断事实真伪，仅负责数据的召回与物理精炼。

---

## 2. 核心业务流程

系统遵循“执行-审计-反馈-升阶”的循环逻辑，最多进行 3 次迭代。

### 2.1 任务分解与动态提示词注入

- **输入接收**：接收原始 Query 和来自 Check Data Agent 的**额外提示词（Extra Hints）**。
    
- **分解逻辑**：利用当前等级的 LLM 将任务拆解为 3 个独立的搜索子目标。
    
- **提示词偏移**：若存在额外提示词（如“侧重 2025 年后的财务数据”），Scrap Agent 必须在分解过程中将其转化为搜索约束指令。
    

### 2.2 多搜索引擎调度

- **初始轮次**：默认调用 **Tavily API** 进行基准搜索。
    
- **反馈迭代轮次**：
    
    - **领域识别**：自动判断 Query 所属领域（如金融、医疗、技术）。
        
    - **动态路由**：从预设引擎池中选择相关度最高的 **3 个搜索引擎**（如：Arxiv + Google Scholar + Semantic Scholar）。
        

---

## 3. 数据处理核心模块

### 3.1 物理去重模块 (Physical Deduplication)

为节省后续处理成本并避免冗余，系统需进行硬去重。

- **URL Set Union**：
    
    - 对所有搜索引擎返回的结果 URL 进行 **Set 合集操作**（`set.union`）。
        
    - 确保进入文本提取阶段的每一个 URL 是唯一的。
        
- **原始文段提取**：对唯一 URL 进行 HTML 解析，提取正文并剔除广告、导航栏等噪声。
    

### 3.2 文本压缩与 MMR 重排 (Semantic Refinement)

- **文本压缩**：将长网页精炼为多个语义完整的文段（Passages），每个文段控制在 200-500 字。
    
- **MMR (Maximal Marginal Relevance) 算法执行**：
    
    - **相关性计算**：计算文段与搜索目标的向量相似度（不作为最终得分）。
        
    - **多样性平衡**：通过惩罚与已选文段相似度高的内容，确保 Top 10 文段覆盖不同的角度。
        
    - **中立性原则**：Scrap Agent 必须保留存在矛盾的信息，由审计 Agent 判断真伪。
        

---

## 4. 模型升阶与迭代机制

Scrap Agent 的推理能力随轮次递增，以应对审计 Agent 提出的更复杂的反馈要求。

|**轮次**|**模型等级**|**典型配置**|**触发条件**|
|---|---|---|---|
|**第一轮**|Level 1 (Base)|GPT-4o-mini / Llama 3|初始查询开始|
|**第二轮**|Level 2 (Pro)|GPT-4o / Claude 3.5 Sonnet|审计 Agent 返回“目标缺失”|
|**第三轮**|Level 3 (Max)|o1-preview / Claude 3 Opus|置信分数持续不达标|

---

## 5. 接口契约与权责定义

### 5.1 严禁行为 (Strict Prohibitions)

- **禁止打分**：Scrap Agent 严禁生成任何形式的 `Confidence Score` 或 `Truthfulness Rating`。
    
- **禁止虚构**：若搜索无结果，需如实向审计 Agent 报告，不得生成幻觉数据。

## 6. 非功能性需求

- **置信度评价依赖**：系统必须在收到审计 Agent 的 `is_satisfied: false` 指令及 `confidence_score` 后，方可启动下一轮搜索。
    
- **MMR 效率**：语义去重与重排的总耗时需控制在 5s 以内。
    
- **容错处理**：若垂直搜索引擎 API 调用失败，自动回退（Fallback）至 Google/Bing 通用搜索。

# 产品需求文档 (PRD)：Check Data Agent (CDA) 智能核查模块

**项目代号：** Project Sentinel (哨兵)

**版本：** V1.1 (动态原子化核实版)

**当前日期：** 2026年2月4日

---

## 1. 文档概述

### 1.1 目的

定义 Check Data Agent 的功能规格，旨在解决 Scrap Agent 检索回来的内容虽具有“语义相关性”但存在“逻辑/事实偏差”的问题。

### 1.2 核心痛点

- **年份陷阱：** 搜索 2026 年，返回 2025 年。
    
- **主体混淆：** 搜索 A 公司，返回了 B 公司的对比文段。
    
- **模棱两可：** 返回的内容是预测（Forecast）而非实绩（Actuals）。
    

---

## 2. 模块架构与工作流

### 2.1 整体链路

1. **原子化拆解 (SLM):** 将目标拆解为不可退让的“硬约束”。
    
2. **硬性过滤 (Custom Check):** 物理层比对关键词与实体。
    
3. **语义定罪 (DeepEval):** 运行 ClaimCheck，判定逻辑支撑。
    
4. **指令生成 (Feedback):** 判定失败时，动态重写抓取策略。
    

---

## 3. 功能需求

### 3.1 目标原子化模块 (Atomic Deconstructor)

**执行者：** Small Language Model (SLM)

- **需求：** 将用户搜索目标转化为结构化清单。
    
- **输出样例：**
    
    - **核心实体 (Subject):** Nvidia
        
    - **时间戳 (Time Constraint):** 2026-Q1
        
    - **关键指标 (Metric):** AI GPU Revenue
        
    - **禁区 (Negative Constraints):** 2024, 2025, Projection data.
        

### 3.2 自定义约束校验 (Custom Constraint Guard)

**执行者：** 逻辑判定脚本 (Logic Script)

- **需求：** 在进入大模型评测前，进行预审。
    
- **校验矩阵：**
    

|**检查项**|**校验逻辑**|**失败动作**|
|---|---|---|
|**存在性检查**|文段是否包含目标年份/主体词？|直接判定 `FAIL`，进入反馈。|
|**实体密度**|目标词在压缩文段中的出现频率与距离。|低于阈值标记为 `LOW_CONFIDENCE`。|
|**关键词匹配**|是否包含“预测”、“幻觉词”、“旧版本”？|标记为 `SUSPICIOUS`。|

### 3.3 深度评估模块 (DeepEval Core)

**执行者：** DeepEval 框架 (`FactualityMetric` / `GEval`)

- **需求：** 基于 SLM 提取的原子 Claim 进行逻辑对账。
    
- **判定逻辑公式：**
    
    $$Score = \frac{Verified\_Claims}{Total\_Required\_Claims}$$
    
- **详细要求：**
    
    - 使用 `FactualityMetric` 核实数据源是否真实支撑目标。
        
    - 利用 `Reasoning` 字段捕获失败的具体逻辑（如“数据为预测值而非实绩”）。
        

### 3.4 动态指令生成 (Feedback Loop)

**执行者：** Check Data Agent (Core Agent)

- **需求：** 当校验不通过时，生成给 Scrap Agent 的补救方案。
    
- **生成规则：**
    
    - 如果是年份错误：生成包含 `-"2024" -"2025"` 的布尔搜索式。
        
    - 如果是信息缺失：提取缺失的 Claim 关键词，作为下一轮搜索的主词。

# 产品需求文档 (PRD)：智能深度报告生成引擎

|**项目名称**|**Writer Agent 自动化撰写模块**|**版本**|**v1.2**|

## 1. 业务概述与目标

### 1.1 背景描述

在多 Agent 协作系统中，前置的搜索 Agent 和分析 Agent 会产生大量碎片化信息。本模块（Writer Agent）旨在将这些带有“置信度评分”的原始语料，转化为结构化、专业且具备逻辑深度的最终报告。

### 1.2 核心目标

- **质量校准**：通过置信度过滤，确保报告的核心论点建立在可靠数据之上。
    
- **透明度标注**：对不确定性信息进行显性标注，辅助决策者判断。
    

---

## 2. 功能需求详情

### 2.1 语料处理逻辑

系统接收来自上游 Agent 的 JSON 格式语料包。处理逻辑如下：

- **高置信度语料 ($\ge 0.85$)**：作为事实依据（Ground Truth），直接进入正文撰写。
    
- **中置信度语料 ($0.50 - 0.84$)**：作为补充论据，需结合上下文进行“多方参考”式描述。
    
- **低置信度语料 ($< 0.50$)**：严格标注，仅作为可能性的探索性内容。
    

---

### 2.2 核心模块架构 (模块化设计)

#### 2.2.1 结构化映射模块

- **功能**：解析用户输入大纲，建立内容索引。
    
- **逻辑**：根据语料中的关键词标签与大纲节点的语义匹配度进行填充。
    

#### 2.2.2 冲突裁决引擎

- 当针对同一关键指标出现不同语料时，采用置信度加权算法。
    
- 计算公式示例：
    
    $$S_{final} = \frac{\sum (Content_i \times Confidence_i)}{\sum Confidence_i}$$
    
    _(注：此公式用于定量数据的加权平滑处理。)_
    

---

## 3. 置信度差异化表现 (报告示例展示)

以下展示在本 PRD 中，Agent 如何根据不同置信度撰写“技术可行性”章节：

### 3.1 技术实现路径

基于高置信度语料，系统将采用 Transformer 架构的变体作为核心生成引擎，结合 RAG（检索增强生成）技术，确保文本生成的流畅性。

### 3.2 性能预估

- **生成速度**：预计在 30 秒内完成 5000 字报告的初稿生成（置信度：0.92）。
    
- **准确率**：事实性错误率控制在 2% 以内（置信度：0.88）。
    
- _⚠️ **[低置信度标注]**：有非正式语料指出，该引擎在处理极端生僻领域（如古代炼金术语）时，准确率可能会下降至 60% 以下。该数据源自单一开源社区讨论，未经大规模 Benchmark 测试，仅供参考。_

## 4. 非功能性需求

|**维度**|**需求描述**|**验收标准**|
|---|---|---|
|**可读性**|消除段落间的跳跃感，增加逻辑连接词。|人工评测流畅度分值 $> 4.0/5.0$|

|**格式化**|支持 Markdown、PDF、Docx 导出。|导出格式层级对齐。|

## 5. 风险与对策

- **风险 1**：所有输入语料置信度均低于 0.3。
    
    - **对策**：Agent 触发预警，在报告开头显著位置提示“本报告基于低可靠性素材生成，请谨慎参考”。
        
- **风险 2**：大纲某个 H2 章节下无任何语料。
    
    - **对策**：保留标题，并标注“该部分暂无相关语料支撑”

# PRD 补充：基于语料库置信度的严苛审计模块

## 1. 输入数据定义 (Input Definition)

Review Agent 的输入现在由四要素组成：

1. **最新报告 ($V_n$)：** Revise Agent 提交的版本。
    
2. **修改清单 ($Manifest$)：** Revise Agent 自述的改动点。
    
3. **权威语料库 ($Corpus$)：** 包含事实、数据及其对应的 **置信度百分比**。
    
4. **历史意见堆栈 ($Stack$)：** 所有前几轮未彻底解决或需要持续监控的指令。
    

---

## 2. 核心审计逻辑：语料库锚定 (Corpus-Based Grounding)

### 2.1 零外部知识原则 (Zero External Knowledge)

- **规则：** Review Agent 必须屏蔽所有训练数据中的外部常识。
    
- **判定：** 如果报告中出现的内容在语料库中找不到出处（哪怕它在现实中是正确的），Review Agent 必须将其标记为“无依据内容”并要求删除。
    

### 2.2 置信度阶梯校准 (Confidence Mapping)

Review Agent 必须强制 Revise Agent 的语气与语料库中的置信度对齐。审计标准如下：

|**语料库置信度 (Conf)**|**报告允许的表达方式**|**Review Agent 审计动作**|
|---|---|---|
|**高 (90% - 100%)**|确定性陈述（“是”、“确定”、“证实”）|若 Revise Agent 使用模糊词，则要求其“说清楚”。|
|**中 (50% - 89%)**|可能性陈述（“可能”、“倾向于”、“大概率”）|若 Revise Agent 说得太死，则要求其“降温”。|
|**低 (< 50%)**|存疑性陈述（“尚不确定”、“数据存疑”、“待证实”）|**禁止强制说清楚**，必须保留不确定性。|
|**未提及 (N/A)**|不允许出现|**强制要求删除**。|

---

## 3. 动态审计流程 (Iteration Loop)

1. **语料库扫描：** 首先解析语料库，建立一个“事实-置信度”映射表。
    
2. **报告一致性对比：**
    
    - 检查 $V_n$ 是否引入了语料库之外的信息。
        
    - 检查 $V_n$ 中每个数据点的语气是否与置信度等级匹配。
        
3. **清单与回归审计：**
    
    - 比对 $Manifest$。
        
    - 检查前几轮已解决的“置信度对齐”问题是否在本轮被意外篡改（回归检测）。
        
4. **输出指令：** 发送包含“事实溯源要求”的新一轮意见书。
    

---

## 4. 审计意见书示例 (Audit Output)

> ### 🔍 语料库一致性审计 (Iteration #4)
> 
> #### 1. 置信度违规项 (Confidence Mismatch)
> 
> - **[位置]：** 第二章《行业增长预期》
>     
> - **[语料库依据]：** 文献 ID #A-12，置信度 **42%**。
>     
> - **[报告现状]：** “行业将迎来 20% 的稳健增长。”
>     
> - **[审计判定]：** **过度承诺。** 语料库置信度低，严禁使用“稳健增长”。请修改为“基于初步迹象推测，可能存在增长空间，但数据置信度较低，需谨慎对待”。
>     
> 
> #### 2. 知识边界违规项 (Out-of-Bounds)
> 
> - **[内容]：** 报告中提到的“主要竞争对手 B 公司的近期裁员消息”。
>     
> - **[审计判定]：** **无依据删除。** 语料库中不包含任何关于 B 公司裁员的信息。即便这是事实，也不得出现在本报告中。
>     
> 
> #### 3. 存疑保留审计 (Ambiguity Preservation)
> 
> - **[确认]：** 上一轮要求的“针对 X 技术的不确定性保留”在本轮得到维持，Revise Agent 未尝试对其进行虚假平滑。
>     

---

## 5. 关键非功能性需求

- **溯源强制性：** Review Agent 的每一条修改意见都必须强制引用语料库中的 `Entry ID`。
    
- **不解释权：** Review Agent 不需要向 Revise Agent 解释为什么要这么改，只需要指出“与语料库置信度不符”。
    
- **抗干扰能力：** 即使 Revise Agent 在清单中辩解“为了逻辑通顺添加了背景信息”，Review Agent 也要铁面无私地将其驳回——**只要语料库没有，报告就不准有。**
    

---

## 6. 验收标准

1. **100% 语料回溯：** 报告中的每一个核心事实都能在语料库中找到对应。
    
2. **语气精准：** 报告的确定性词汇分布与语料库置信度分布呈正相关。
    
3. **零遗忘：** 历史意见集中的“存疑点”在连续 10 轮迭代中依然被正确标记为存疑。


# 产品需求文档 (PRD): Revise Agent

## 1. 定位与目标

**Revise Agent** 是一款基于大语言模型（LLM）的智能校对与改写引擎。它不仅根据用户的直观意见进行修改，更重要的是通过检索**带置信度的语料库**来确保修改内容的专业性、准确性与合规性。

---

## 2. 输入数据定义

Agent 接收三个核心输入：

|**输入项**|**格式**|**说明**|
|---|---|---|
|**原始报告 (Report)**|Markdown / Text|待修改的初始文档。|
|**修改意见 (Feedback)**|列表 (List)|用户提出的具体改进要求，如“第二章数据过时”、“语气太口语化”等。|
|**语料库 (Corpus)**|结构化数据 (JSON/DB)|包含 `Content`（内容）、`Source`（来源）及 `Confidence`（置信度：0.0-1.0）。|

---

## 3. 核心功能需求

### 3.1 修改意见解构与对齐

- **需求描述：** Agent 必须将模糊的修改意见拆解为可执行的原子任务。
    
- **逻辑：** 识别修改意见指向报告的具体段落或数据点。
    

### 3.2 语料库检索与置信度过滤

- **需求描述：** 针对每一条修改意见，Agent 需在语料库中检索相关素材。
    
- **处理规则：**
    
    - **高置信度 ($\ge 0.8$)：** 直接采用语料库内容替换原报告相关描述。
        
    - **中置信度 ($0.5 \le x < 0.8$)：** 参考语料库内容进行改写，并在修改日志中提醒用户核实。
        
    - **低置信度 ($< 0.5$)：** 仅作为风格参考，或跳过检索，优先遵循修改意见。
        

### 3.3 逐条修改逻辑 (Iterative Revision)

- **需求描述：** Agent **禁止**一次性模糊改写，必须针对 Feedback 列表进行逐条处理。
    
- **执行流程：** 1. 读取反馈 $F_n$。
    
    2. 定位报告段落 $P_n$。
    
    3. 匹配语料库 $C_n$。
    
    4. 生成修改后段落 $P'_n$。
    

---

## 4. 输出要求

Agent 的输出必须包含两部分：

### 4.1 修改后的最终报告

- 保持原有的 Markdown 结构。
    
- 确保逻辑连贯，避免因逐条修改导致的上下文断层。
    

### 4.2 修改记录对照表 (Change Log)

为了便于审计，需输出以下格式的表格：

|**序号**|**原始意见 (Feedback)**|**修改动作**|**采用语料及置信度**|**修改理由**|
|---|---|---|---|---|
|1|“更新 2024 营收数据”|将 100 亿改为 125 亿|《2024 财报》 (0.98)|语料库置信度极高，覆盖旧数据。|
|2|“优化开头语气”|将“我觉得”改为“经分析”|N/A|遵循修改意见，提升专业感。|

---

## 5. 工作流逻辑 (Workflow)

1. **Parse:** 解析 Report 结构，建立索引。
    
2. **Retrieve:** 根据 Feedback 关键词在 Corpus 中进行语义搜索。
    
3. **Synthesize:** * 若 Feedback 与 Corpus 一致，执行强化修改。
    
    - 若 Feedback 与 Corpus 冲突，以**高置信度语料**为准，并向用户报告冲突。
        
4. **Refine:** 进行最后的润色，确保全文语气统一。
    

---

## 6. 非功能性需求

- **原子性：** 每一条修改意见必须有明确的去向（已修改/因置信度不足未修改）。
    
- **追溯性：** 所有的修改必须能追溯到具体的语料来源。
    
- **格式保留：** 不得擅自改变非修改区域的排版格式。
